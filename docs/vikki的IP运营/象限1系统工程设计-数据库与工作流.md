# 象限1痛点系统工程设计

> 从方法论到系统：数据库架构 + 工作流 + 武器库 + 行业适配

---

## 🎯 系统设计理念

### 核心原则

**1. 数据资产化**
```
每一次调研 → 沉淀到数据库
每一个痛点 → 标签化存储
每一个案例 → 可检索复用

目标：
- 痛点数据库越用越丰富
- 从0到1：靠调研
- 从1到N：靠积累
- 从N到∞：靠AI挖掘
```

**2. 流程标准化**
```
工作流 = 输入 → 处理 → 输出 → 积累

每日工作流：数据扫描 → 痛点提取 → 入库
每周工作流：深度调研 → 分析提炼 → 更新库
每月工作流：数据分析 → 策略调整 → 优化库
```

**3. 工具武器化**
```
武器库 = 方法 + 话术 + 模板 + 检查清单

每个场景都有：
- SOP（标准操作流程）
- 话术库（可直接使用）
- 模板（Excel/Notion/文档）
- 检查清单（确保不遗漏）
```

**4. 持续进化**
```
系统 = 静态框架 + 动态数据

框架设计：一次设计，长期使用
数据积累：持续迭代，越来越精准
AI辅助：自动化程度逐步提升
```

### 1.2 核心表结构设计

#### 表1：painpoint_master（痛点主库）

**作用：** 存储所有识别出的痛点

```sql
CREATE TABLE painpoint_master (
    -- 基础信息
    id VARCHAR(50) PRIMARY KEY,  -- 痛点唯一ID
    painpoint_name VARCHAR(200) NOT NULL,  -- 痛点名称
    painpoint_type ENUM('即时性', '持续性', '隐性') NOT NULL,  -- 痛点类型

    -- 痛点描述
    description TEXT,  -- 详细描述
    user_scenario TEXT,  -- 用户场景
    user_quote TEXT,  -- 用户原话

    -- 分类标签
    industry VARCHAR(100),  -- 所属行业
    sub_industry VARCHAR(100),  -- 细分领域
    user_segment VARCHAR(100),  -- 用户细分

    -- 痛点特征（即时性专属）
    urgency_level ENUM('高', '中', '低'),  -- 紧迫程度
    time_sensitivity VARCHAR(50),  -- 时效性（如"24小时内"）
    trigger_scene TEXT,  -- 触发场景

    -- 痛点特征（持续性专属）
    frequency VARCHAR(50),  -- 发生频率（如"每周"）
    duration VARCHAR(50),  -- 持续时长（如"3个月"）
    recurrence_pattern TEXT,  -- 重复模式

    -- 痛点特征（隐性专属）
    visibility_level ENUM('完全隐性', '半隐性', '准隐性'),  -- 隐性程度
    shame_factor TEXT,  -- 羞耻因素
    social_pressure TEXT,  -- 社会压力
    internal_conflict TEXT,  -- 内心冲突

    -- 数据来源
    source_type ENUM('搜索数据', '用户访谈', '社群潜伏', '评论区', '咨询记录', '其他'),
    source_platform VARCHAR(50),  -- 来源平台
    source_link TEXT,  -- 来源链接
    discovered_date DATE,  -- 发现日期
    discovered_by VARCHAR(50),  -- 发现人

    -- 验证信息
    is_validated BOOLEAN DEFAULT FALSE,  -- 是否已验证
    validation_method VARCHAR(100),  -- 验证方法
    validation_score INT,  -- 验证分数（0-100）

    -- 商业价值
    market_size ENUM('大', '中', '小'),  -- 市场规模
    urgency_score INT,  -- 紧迫度评分（0-100）
    frequency_score INT,  -- 频次评分（0-100）
    commercial_value INT,  -- 商业价值评分（0-100）

    -- 解决方案匹配
    product_match_level ENUM('强相关', '中相关', '弱相关'),  -- 产品匹配度
    solution_difficulty ENUM('容易', '中等', '困难'),  -- 解决难度

    -- 竞争分析
    competitor_coverage ENUM('蓝海', '红海', '竞争激烈'),  -- 竞争程度
    content_saturation ENUM('低', '中', '高'),  -- 内容饱和度

    -- 效果追踪
    times_used INT DEFAULT 0,  -- 使用次数
    total_views INT DEFAULT 0,  -- 总浏览量
    total_conversions INT DEFAULT 0,  -- 总转化数
    avg_conversion_rate DECIMAL(5,2),  -- 平均转化率

    -- 状态管理
    status ENUM('待验证', '已验证', '使用中', '已失效', '已归档'),
    priority ENUM('高', '中', '低'),  -- 优先级

    -- 时间戳
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    -- 备注
    notes TEXT,  -- 备注
    tags JSON  -- 自定义标签（JSON数组）
);
```

### 1.4 数据录入流程

#### 流程1：即时性痛点录入

```
步骤1：数据扫描（AI）
├── 输入：平台（百度/小红书/抖音等）
├── 输出：候选痛点列表（包含关键词、搜索量、趋势）
└── 工具：痛点扫描Agent

步骤2：人工筛选
├── 检查：是否真实痛点
├── 检查：产品是否能解决
├── 检查：商业价值
└── 决策：录入 or 丢弃

步骤3：信息补全
填写字段：
├── painpoint_name：手机内存不足
├── painpoint_type：即时性
├── description：用户手机存储空间即将满，影响拍照、下载
├── user_scenario：准备拍照时提示内存不足
├── urgency_level：高
├── time_sensitivity：立即（<1小时）
├── trigger_scene：拍照、下载、更新APP
├── industry：消费电子
├── source_type：搜索数据
├── source_platform：百度
├── discovered_date：2026-01-07
└── commercial_value：85

步骤4：自动关联
系统自动：
├── 关联行业标签
├── 关联用户画像
├── 生成唯一ID
└── 计算优先级
```

#### 流程2：持续性痛点录入

```
步骤1：问题记录（日常）
每天记录用户问题到临时表：
├── 日期
├── 用户昵称/来源
├── 问题原文
└── 问题分类

步骤2：周度统计（每周日）
统计本周高频问题：
├── 哪个问题被问>5次？
├── 哪个问题不同用户都问？
└── 哪个问题你回答了但用户还不会？

步骤3：月度提炼（每月底）
分析持续性：
├── 连续3周都被问的问题
├── 每月都出现的问题
└── 提炼为持续性痛点

步骤4：录入数据库
填写字段：
├── painpoint_type：持续性
├── frequency：每周
├── duration：3个月
├── recurrence_pattern：每次选题都纠结
└── source_type：咨询记录
```

#### 流程3：隐性痛点录入（最复杂）

```
步骤1：调研执行
使用武器库中的方法：
├── 潜伏社群（1个月）
├── 深度访谈（5-10人）
└── 评论区考古（50条内容）

步骤2：洞察提炼
记录模板：
├── 用户原话
├── 背后恐惧
├── 社会压力
├── 内心冲突
└── 合理化需求

步骤3：验证确认
交叉验证：
├── 至少3个来源提到相同痛点
├── 深度访谈中被确认
└── 评论区有情绪共鸣

步骤4：录入数据库
填写字段：
├── painpoint_type：隐性
├── visibility_level：完全隐性
├── shame_factor：怕被说不专业
├── social_pressure：行业认为用AI=偷懒
├── internal_conflict：想用AI vs 想保持专业形象
└── source_type：社群潜伏 + 深度访谈
```

### 1.5 数据查询场景

#### 场景1：按行业查找痛点

```sql
-- 查找美妆行业的所有即时性痛点，按商业价值排序
SELECT
    painpoint_name,
    description,
    commercial_value,
    urgency_level,
    discovered_date
FROM painpoint_master
WHERE industry = '美妆护肤'
    AND painpoint_type = '即时性'
    AND status = '已验证'
ORDER BY commercial_value DESC
LIMIT 20;
```

#### 场景2：查找未使用的高价值痛点

```sql
-- 查找商业价值>80但还没用过的痛点
SELECT
    pm.painpoint_name,
    pm.painpoint_type,
    pm.commercial_value,
    pm.industry,
    COUNT(pc.id) as content_count
FROM painpoint_master pm
LEFT JOIN painpoint_content pc ON pm.id = pc.painpoint_id
WHERE pm.commercial_value > 80
    AND pm.status = '已验证'
GROUP BY pm.id
HAVING content_count = 0
ORDER BY pm.commercial_value DESC;
```

#### 场景3：分析痛点效果

```sql
-- 分析某个痛点的内容效果
SELECT
    pm.painpoint_name,
    COUNT(pc.id) as content_count,
    AVG(pc.conversion_rate) as avg_conversion_rate,
    SUM(pc.conversions) as total_conversions,
    SUM(pc.revenue) as total_revenue
FROM painpoint_master pm
JOIN painpoint_content pc ON pm.id = pc.painpoint_id
WHERE pm.id = 'painpoint_001'
GROUP BY pm.id;
```

#### 场景4：发现隐性痛点趋势

```sql
-- 查找最近3个月发现的隐性痛点，按发现日期分组
SELECT
    DATE_FORMAT(discovered_date, '%Y-%m') as month,
    COUNT(*) as painpoint_count,
    AVG(commercial_value) as avg_value
FROM painpoint_master
WHERE painpoint_type = '隐性'
    AND discovered_date >= DATE_SUB(CURDATE(), INTERVAL 3 MONTH)
GROUP BY month
ORDER BY month DESC;
```

## 🔄 二、工作流系统设计

### 2.1 每日工作流（数据扫描与记录）

**时间：每天30分钟**

```
┌─────────────────────────────────────────┐
│       每日痛点扫描工作流 (30min)          │
└─────────────────────────────────────────┘

【8:00-8:15】即时性痛点扫描（15min）
├── Step 1：打开痛点扫描Agent
│   ├── 输入行业关键词
│   ├── 选择平台：百度+小红书+抖音
│   └── 时间范围：近24小时
│
├── Step 2：AI自动扫描
│   ├── 提取热搜词
│   ├── 识别时间紧迫词
│   ├── 过滤无关词
│   └── 输出候选痛点列表（20-50个）
│
├── Step 3：人工快速筛选（10min）
│   ├── 浏览列表
│   ├── 标记：✅录入 / ⚠️待验证 / ❌丢弃
│   └── 标准：
│       - 搜索量>1000
│       - 产品能解决
│       - 未重复
│
└── Step 4：批量录入（5min）
    ├── 选中✅的痛点
    ├── 点击"批量录入"
    ├── AI自动填充基础字段
    └── 状态：待验证

【8:15-8:25】持续性痛点记录（10min）
├── Step 1：打开问题记录表
│
├── Step 2：记录昨天的用户提问
│   来源：
│   ├── 私信（3条）
│   ├── 评论区（5条）
│   ├── 社群（8条）
│   └── 咨询（2条）
│
├── Step 3：记录格式
│   │ 日期  │ 来源  │ 用户ID │ 问题原文 │ 分类 │
│   │ 1/7  │ 私信  │ user01 │ 怎么选题 │ 选题 │
│
└── 自动统计：
    ├── 本周高频问题Top10
    └── 预警：某问题被问>5次

【8:25-8:30】隐性痛点线索记录（5min）
├── 浏览昨天潜伏社群的聊天记录
├── 标记可疑隐性痛点
│   示例："又被老公说乱花钱了"
│   标记：💡隐性痛点线索
└── 加入调研清单（周末深入挖掘）
```

- 痛点扫描Agent（自动化）
- Excel/Notion问题记录表（手动）
- 社群聊天记录收集工具（半自动）

### 2.2 每周工作流（深度验证与调研）

```
┌─────────────────────────────────────────┐
│      每周痛点深度工作流 (3h)             │
└─────────────────────────────────────────┘

【9:00-10:00】持续性痛点提炼（1h）

│   ├── 统计问题频次
```
