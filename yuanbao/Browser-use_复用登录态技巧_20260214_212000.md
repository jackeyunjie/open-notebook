# Browser-use 复用登录态技巧

## 总结

> Browser-use Skill 在使用时有一个常见痛点：每次任务都会新开浏览器环境，导致登录态、Cookie丢失，需要反复登录。解决方法非常简单——使用 `--browser-real` 模式调用本机真实浏览器，直接复用已有的登录会话和权限信息，大幅提升效率。

---

## 正文

### 问题现象

使用 Browser-use Skill 抓取网页时，遇到需要登录的网站（小红书、X/Twitter、后台系统等）：

```
❌ 错误用法（默认模式）
- 每次任务新开浏览器环境
- Cookie、Session、登录态全部丢失
- 需要反复扫码/登录
- 真正浪费时间的是登录，不是抓数据
```

### 解决方案

```
✅ 正确用法（--browser-real 模式）
- 调用本机真实浏览器（Chrome/Edge）
- 复用已有的登录态和Cookie
- 直接访问需要权限的页面
- 抓取效率提升10倍+
```

---

### 使用方法

在对话中明确指定使用 `--browser-real` 模式：

```bash
# 示例1：抓取小红书数据
"用 browser-use --browser-real 模式帮我抓取小红书上关于'装修'的热门笔记"

# 示例2：抓取X/Twitter
"用 browser-use --browser-real 模式获取我关注的AI领域最新推文"

# 示例3：访问后台系统
"用 browser-use --browser-real 模式登录后台，导出昨天的订单数据"
```

---

### 原理说明

| 模式 | 浏览器环境 | 登录态 | 适用场景 |
|------|-----------|--------|---------|
| **默认模式** | 新开虚拟浏览器 | ❌ 不保留 | 公开页面、无需登录 |
| **--browser-real** | 调用本机真实浏览器 | ✅ 完全复用 | 需要登录、个人账号 |

**--browser-real 模式优势：**
- 复用本机浏览器的所有Cookie和Session
- 保留已登录账号的权限信息
- 支持需要二次验证的网站
- 可以使用浏览器插件（如广告拦截）

---

### 前置条件

使用 `--browser-real` 模式前，确保：

1. **本机已安装 Chrome 或 Edge 浏览器**
2. **已在浏览器中登录目标网站**
   - 小红书：已扫码登录
   - X/Twitter：已账号登录
   - 后台系统：已认证通过
3. **浏览器未开启"退出时清除数据"**

| "需要复杂的Cookie管理" | 直接复用浏览器，无需手动管理 |

```
3. 定期维护

```
